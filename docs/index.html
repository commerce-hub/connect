<commerce-hub-connect>

<commerce-hub-connect> is the CommerceHub Connect Web SDK widget. It lets a merchant host embed CommerceHub payment-source flows as a standards-based web component while keeping CommerceHub authentication and interaction creation on the host backend.

What the SDK handles

  • OTP verification and resend for CommerceHub interactions.
  • Saved payment-source selection, attach, verification, and deletion.
  • Card entry with sensitive value masking before source creation.
  • Manual ACH entry and Pay by Bank provider handoff.
  • Required-info collection for sources that need additional verification.
  • Native account balance lookup when native sources are present.
  • Merchant-safe styling and copy customization through designConfig.

Integration shape

The host application creates or retrieves an interaction, passes the resulting InteractionResponse into the widget, and listens for onSelect when the shopper finishes with a selected payment source. The host then completes the merchant interaction with GET /merchant/interaction/{interactionId}/complete; the widget does not call that endpoint internally.

Assign the required interactionResponse and optional designConfig as DOM properties in every framework. Do not pass object-valued SDK inputs as HTML attributes; attributes can only carry strings. The base-url attribute is available only for the string baseURL value.

For SDK-managed operations before completion, the widget also emits success-only events: onListPaymentMethods, onAddPaymentMethod, onVerifyPaymentMethod, onDeletePaymentMethod, and onTransfer.

The host response is bootstrap data. When a new interactionResponse.interactionId is assigned, the widget immediately calls GET /interaction and renders saved or selected payment methods only from that server-fetched response. Host-provided paymentSources, selectedSource, billing addresses, and status fields are not authoritative widget state.


import 'commerce-hub-connect-sdk';

const widget = document.querySelector('commerce-hub-connect');

if (widget) {
  widget.interactionResponse = await fetch('/api/commercehub/interaction').then(
    (response) => response.json()
  );
  widget.designConfig = {
    colors: {
      primary: '#084acc',
      text: '#101828',
      border: '#cbd5e1',
      widgetBackground: '#ffffff',
    },
  };
  widget.baseURL = 'QA';
  widget.addEventListener('onSelect', async (event) => {
    const selectedPaymentSource = event.detail;
    const completedPaymentSource = await fetch(
      `/api/commercehub/interaction/${encodeURIComponent(
        widget.interactionResponse.interactionId
      )}/complete`,
      {method: 'GET'}
    ).then((response) => response.json());

    // Store or reference selectedPaymentSource in host-owned state as needed.
    // Continue the host checkout flow with completedPaymentSource.
  });
}

Request routing

SDK-managed CommerceHub calls always send interactionResponse.interactionId as the x-interaction-id header. When interactionResponse.oauth.access_token is present, they also send Authorization: Bearer <token>. If the interaction id is missing, the widget does not call the API and keeps the current screen stable.

If interactionResponse itself is missing, the widget renders a configuration error and blocks shopper payment actions until the host assigns it.

Gateway OAuth from the host response is preserved for SDK-managed calls when the refreshed GET /interaction response omits it.

The widget uses the PROD preset by default. Set .baseURL to a different preset or custom origin before the shopper starts the flow only when you need to override production routing:


widget.baseURL = 'PROD';
widget.baseURL = 'QA';
widget.baseURL = 'https://api.example.com';

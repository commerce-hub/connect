<commerce-hub-connect>

<commerce-hub-connect> is the Commerce Hub Connect Web SDK widget. It lets a merchant host embed Commerce Hub payment-source flows as a standards-based web component while keeping Commerce Hub authentication and interaction creation on the host backend.

What the SDK handles

  • OTP verification and resend for Commerce Hub interactions.
  • Saved payment-source selection, attach, verification, and deletion.
  • Card entry with sensitive value masking before source creation.
  • Manual ACH entry and Pay by Bank provider handoff.
  • Required-info collection for sources that need additional verification.
  • Native account balance lookup when native sources are present.
  • Merchant-safe styling and copy customization through designConfig.

Integration shape

The host application creates or retrieves an interaction, passes the resulting InteractionResponse into the widget, and listens for onSuccess when the shopper finishes with a selected payment source. The host then completes the merchant interaction with POST /merchant/interaction/complete; the widget does not call that endpoint internally.


import 'commerce-hub-connect-sdk';

const widget = document.querySelector('commerce-hub-connect');

if (widget) {
  widget.interactionResponse = await fetch('/api/commercehub/interaction').then(
    (response) => response.json()
  );
  widget.baseURL = 'QA';
  widget.addEventListener('onSuccess', async (event) => {
    const selectedPaymentSource = event.detail;
    const completedPaymentSource = await fetch(
      '/api/commercehub/interaction/complete',
      {
        method: 'POST',
        headers: {'Content-Type': 'application/json'},
        body: JSON.stringify({value: widget.interactionResponse.interactionId}),
      }
    ).then((response) => response.json());

    // Store or reference selectedPaymentSource in host-owned state as needed.
    // Continue the host checkout flow with completedPaymentSource.
  });
}

Request routing

SDK-managed Commerce Hub calls always send interactionResponse.interactionId as the x-interaction-id header. When interactionResponse.oauth.access_token is present, they also send Authorization: Bearer <token>. If the interaction id is missing, the widget does not call the API and keeps the current screen stable.

The widget uses the PROD preset by default. Set .baseURL to a different preset or custom origin before the shopper starts the flow only when you need to override production routing:


widget.baseURL = 'PROD';
widget.baseURL = 'QA';
widget.baseURL = 'https://api.example.com';
